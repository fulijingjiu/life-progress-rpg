# AI 实现执行 Loop

> 用途：交给编码 AI，自主完成 life-progress-rpg v0.1 MVP  
> 状态：可执行指令  
> 权威范围：[MVP 规划 v0.1](./milestone-v1.md)

## 任务

你是 life-progress-rpg 的主实现工程师。你的任务不是只提出方案，而是持续实现、测试和修复，直到 v0.1 MVP 的全部 P0 验收项通过。

项目根目录：

```text
/home/root1/下载/tools/tools/life-progress-rpg
```

## 二、范围约束

只实现 v0.1 P0，以及完成 P0 所必需的基础设施。

禁止自行实现：

- 账号和登录
- 云同步或多设备同步
- 公开分享和社交系统
- 公开排行榜
- 正式成就或 XP 系统
- 推送通知
- 周报、月报和年度报告
- 深度 AI 对话
- AI 人生规划
- AI 医疗、心理、法律或财务建议
- AI 预测和虚假精确百分比
- 微服务、Redis、任务队列和 PostgreSQL
- 文档没有要求的大规模重构

统一数据口径：

- 出生信息只保存年份。
- 人生进度是估算值，必须显示“估算”。
- 用户可关闭人生进度展示。
- 心情固定为 1～5。
- 能量固定为整数 0～10。
- 每个本地日期最多一条记录。
- 日期以用户本地时区为准。
- 默认使用本地当日回应。
- AI 必须由用户明确启用。
- 真实 AI 只能经过服务端代理。
- 浏览器端不得保存供应商 API Key。
- AI 失败不得影响记录保存。

## 三、启动检查

修改代码前：

1. 阅读仓库中的 `AGENTS.md`、`README.md`、`package.json` 和相关配置。
2. 运行 `git status --short`，识别并保留用户已有改动。
3. 根据 `package.json` 确认已有脚本、依赖和项目结构。
4. 运行现有测试、类型检查和构建，记录初始失败。
5. 搜索是否已有数据类型、IndexedDB、状态管理、路由、Onboarding、Record、Settings、History 和 AI 接口。
6. 优先扩展已有结构，避免创建重复模块。
7. 创建或更新 `docs/planning/implementation-status.md`。

`implementation-status.md` 至少记录：

- 当前阶段
- 已完成验收项
- 未完成验收项
- 当前失败测试
- 已知技术债务
- 下一轮最小垂直切片
- 实际运行的验证命令及结果

## 四、实施阶段

### 阶段 A：工程骨架、基础模型和存储

先阅读[代码结构与技术栈边界](../tech/code-structure.md)，只创建当前切片需要的 `app`、`domain`、`data`、`features`、`services`、`shared` 和 `test` 目录。不一次性生成空目录，不创建没有构建入口的 `server/`。

实现：

- `UserSettings`
- `LifeRecord`
- 运行时数据校验
- Dexie/IndexedDB Schema
- `[userId+localDate]` 唯一约束
- Schema 版本与迁移
- Repository CRUD；Zustand 只处理短生命周期 UI/流程状态
- 本地日期工具
- 基础单元测试

验收：

- 心情只能是 1～5。
- 能量只能是 0～10 整数。
- 同日重复保存会更新原记录。
- 刷新页面后数据不丢失。
- 无效数据不能进入数据库。
- 数据库升级不会静默丢失数据。
- Domain 不依赖 React/Dexie，页面不直接访问数据库。
- Dexie 记录没有被复制到 Zustand 或 TanStack Query。

### 阶段 B：首次引导和设置

实现：

- 可跳过的昵称
- 出生年份
- 可调整预期年限
- 人生进度显示开关
- AI 同意开关，默认关闭
- 行为分析同意开关，默认关闭
- 主题设置
- 表单校验和错误提示

验收：

- 60 秒内可以完成引导。
- 不要求完整生日。
- 不启用 AI 也能使用全部核心记录功能。
- 关闭进度后首页不再显示人生进度。
- 设置变化可以持久化。

### 阶段 C：首页和每日记录

实现：

- 用户摘要
- 估算人生进度
- “估算”说明
- 今日记录入口
- 心情选择器
- 能量滑块
- 标签选择
- 可选备注
- 保存中、成功和失败状态
- 离线保存

验收：

- 三步内完成记录。
- 同日再次进入时编辑原记录。
- 保存成功后立即看到反馈。
- AI 不可用时仍保存成功。
- Reduce Motion 下不播放强动画。

### 阶段 D：历史记录

实现：

- 记录列表
- 空状态
- 查看详情
- 编辑记录
- 删除记录
- 删除二次确认
- 日期排序

验收：

- 最新记录排在前面。
- 编辑不会创建重复记录。
- 删除后数据库和 UI 同步更新。
- 空数据和损坏数据有明确反馈。

### 阶段 E：当日回应系统

先按[内容策略](../design/content-strategy.md)实现确定性的本地回应：

- 优先回应备注中的明确事件，其次是标签或当日状态。
- 只有一条记录时不使用“趋势、经常、总是、通常”。
- 信息足够时最多提出一个开放问题。
- 连续三次不使用相同开头。
- 不诊断、预测、评价人格、声称因果或输出虚假百分比。
- 为回应提供有帮助、没帮助和事实不准确反馈。
- 回应重新生成或来源改变时清空旧反馈。

为本地规则添加表驱动测试，覆盖纯数值、长短备注、字段矛盾、重复内容和敏感输入。

如果项目已有安全的服务端运行环境，再实现可选 AI 代理；否则只实现接口、本地规则和测试替身，禁止在前端放置密钥。

AI 代理要求：

- 未同意时拒绝调用。
- 请求 Schema、长度、限流和超时校验。
- 使用固定系统提示和结构化输出。
- 检查输出事实是否存在于允许输入。
- 日志不包含记录正文或完整模型输出。
- 失败时保留本地回应。
- 模型和密钥只由服务端配置。

验收：

- 保存后立即显示有事实依据的本地回应。
- 默认模式不发起 AI 网络请求。
- 未同意时不发送用户内容。
- AI 超时、越权或返回非法数据时自动回退。
- 三类反馈能够保存、导出和重新导入。
- 记录保存不依赖 AI。
- 前端构建产物中不存在 API Key。

### 阶段 F：数据管理

实现：

- JSON 导出
- JSON 导入前预览
- Schema 和版本校验
- 单事务导入
- 冲突处理
- 清空数据
- 清空二次确认

验收：

- 导出的数据能够重新导入。
- 非法 JSON 不会破坏现有数据。
- 导入失败不会留下半写入状态。
- 清空前有明确的不可恢复提示。
- 用户不注册账号也能控制自己的数据。

### 阶段 G：工程质量和可访问性

完成：

- 键盘操作和焦点管理
- 正文对比度和 44×44 最小触摸区域
- Reduce Motion
- 加载、错误、空状态、离线和恢复
- 未处理异常检查
- 数据迁移测试
- 核心流程端到端测试

### 阶段 H：视觉、交互与内容精修

以 `docs/design/quality-bar.md` 为强制验收表，逐页完成：

- 明确唯一主任务和信息层级
- 统一间距、排版、色彩、图标、圆角和阴影
- 删除过量卡片、渐变、发光、Emoji 和装饰噪音
- 使用真实中文短、长、空、异常内容检查布局
- 覆盖 AI 未同意、生成中、失败回退和离线状态
- 检查 360×800、390×844、768×1024 和 1440×900
- 检查触摸、键盘、返回、重复提交和未保存内容恢复
- 逐条精简按钮、说明、成功、错误和空状态文案
- 可用时保存关键页面与状态截图并复核

无法运行浏览器或截图时，把“人工视觉验收”明确记录为未完成，不得仅凭代码宣称界面美观通过。

## 五、持续执行 Loop

重复以下循环，直到满足终止条件。

### LOOP START

#### 1. 读取当前状态

- 运行 `git status --short`。
- 阅读 `implementation-status.md`。
- 获取当前测试、类型检查和构建结果。
- 对照 MVP 验收表找出未完成项。

#### 2. 选择任务

- 选择最高优先级的一个未完成 P0 或设计质量验收项。
- 将其拆成能够独立验证的最小垂直切片。
- 用户可见切片必须在同一轮完成适用的视觉、交互、内容和响应式状态，不能默认留到“最后美化”。
- 每轮只处理一个主要切片。
- 不重复已经完成且验证通过的工作。

#### 3. 实现

- 先阅读相关现有代码和测试。
- 采用最小必要修改。
- 保留用户已有改动。
- 不修改无关模块。
- 同步添加必要测试。
- 不用 TODO、静态成功结果、假数据或空壳界面冒充完成。

每一轮必须产生至少一种有效结果：

- 可验证的代码改动；或
- 明确的失败根因和验证证据；或
- 消除一个已确认的阻塞条件。

禁止只更新计划后进入下一轮。

#### 4. 局部验证

- 运行相关单元测试。
- 运行相关组件或集成测试。
- 运行类型检查。
- 检查本轮差异。
- 修复本轮引入的问题。

#### 5. 全局验证

根据 `package.json` 中真实存在的脚本运行：

- lint
- typecheck
- test
- build
- 必要的端到端测试

不得声称执行了不存在或实际未运行的命令。

#### 6. 失败处理

先把失败分类为：

- 实现缺陷
- 测试缺陷
- 原有失败
- 环境问题
- 外部凭据或服务缺失

对于实现或测试缺陷：

1. 定位根因。
2. 修复根因。
3. 重新运行最小相关测试。
4. 再运行受影响的全局检查。
5. 最多连续尝试三种有实质差异的修复方案。

禁止：

- 删除失败测试
- 跳过关键测试
- 放宽类型来掩盖错误
- 使用 `any` 绕过核心模型
- 注释掉安全校验
- 静态返回成功结果伪装功能完成

#### 7. 更新状态

更新 `implementation-status.md`：

- 本轮修改
- 验证命令和结果
- 新发现问题
- 已完成验收项
- 下一轮任务

#### 8. 判断是否继续

如果仍有安全、可执行的 P0 工作，立即开始下一轮，不等待用户确认。

### LOOP END

## 六、阻塞规则

以下情况不算阻塞，应自行处理：

- 类型错误
- 测试失败
- 构建失败
- 缺少内部工具函数
- 文档中可以合理推断的小细节
- 可以使用本地规则替代的 AI 服务
- 可以通过现有代码确定的设计问题

只有以下情况可以停止：

- 必需外部凭据不存在，并且没有安全的本地替代方案。
- 需要执行任务范围外的破坏性操作。
- 用户已有改动与必需实现直接冲突，且无法安全合并。
- 权威文档存在会造成完全不同产品结果的不可消解冲突。
- 同一个阻塞条件经过至少三轮不同方案仍无法解决。

缺少真实 AI 凭据时，不得停止整个 MVP。继续完成本地当日回应、代理接口、测试替身和失败回退，只把供应商联调记录为外部待办。

## 七、终止条件

只有同时满足以下条件才能结束：

- MVP P0 功能全部实现。
- MVP 功能验收全部通过。
- 类型检查通过。
- 测试通过。
- 生产构建通过。
- 核心端到端流程通过。
- 前端构建产物中不存在供应商 API Key。
- 默认模式不会发送用户记录。
- AI 失败不影响本地保存。
- 所有用户可见页面满足 `docs/design/quality-bar.md` 的适用项。
- 已检查规定视口和关键状态，并保存截图证据；无法执行时明确标记待人工视觉验收。
- 界面文案具体、准确、友好，成功、空状态、错误和恢复内容完整。
- 没有新增失效文档链接。
- `git diff --check` 通过。
- `implementation-status.md` 已更新为完成。
- 最终差异中不存在无关代码改动。

如果任一 P0 验收项仍为 FAIL，不得声称完成，应继续 Loop。

## 八、最终交付格式

结束时输出：

1. 完成的功能。
2. 关键架构决定。
3. 修改的主要文件。
4. 实际运行的验证命令及结果。
5. 尚未完成的外部事项。
6. 不阻止 v0.1 的已知技术债务。
7. MVP 验收表逐项 PASS/FAIL 和对应证据。
8. 视觉、交互、响应式、无障碍和内容质量的复核证据。
9. 尚待人工视觉验收的页面或状态。
10. `git status --short` 和最终差异摘要。
